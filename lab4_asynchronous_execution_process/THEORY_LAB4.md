# Лабораторная работа №4: асинхронный исполнитель задач

## Цель работы

В 4-й лабораторной работе проект из лабораторной №3 расширен до асинхронной системы обработки задач. Очередь из 3-й лабораторной остается источником данных: она по-прежнему лениво получает задачи из генераторов и JSONL-файлов. Новый слой отвечает за асинхронную постановку задач в очередь, выбор подходящего обработчика и выполнение задач без блокировки event loop.

## Что добавлено по сравнению с лабораторной №3

### Асинхронная очередь

Добавлен класс `AsyncTaskQueue`, построенный поверх `asyncio.Queue`.

Он предоставляет асинхронные методы:

- `put()` - добавить задачу в очередь;
- `get()` - получить следующую задачу;
- `task_done()` - отметить завершение обработки;
- `join()` - дождаться обработки всех задач;
- `close()` - отправить служебные сигналы завершения воркерам.

Также очередь умеет загружать задачи из обычного iterable-источника через `load_from_iterable()`. Получение следующего элемента выполняется через `asyncio.to_thread()`, поэтому потенциально блокирующий синхронный источник не останавливает event loop.

### Контракт обработчика

Добавлен `TaskHandler` - runtime-проверяемый `Protocol`.

Обработчик должен реализовать:

- `can_handle(task)` - проверка, подходит ли обработчик для задачи;
- `handle(task)` - асинхронная обработка задачи.

Такой контракт позволяет добавлять новые обработчики без изменения исполнителя. Исполнитель работает не с конкретными классами, а с поведением.

### Асинхронный исполнитель

Добавлен `AsyncTaskExecutor`.

Он:

- принимает список обработчиков;
- проверяет, что каждый обработчик соответствует `TaskHandler`;
- запускает несколько асинхронных воркеров;
- выбирает первый подходящий обработчик для каждой задачи;
- переводит задачу в статус `In Progress`, затем в `Done`;
- централизованно логирует ошибки;
- не прерывает обработку всей очереди из-за одной ошибочной задачи.

Исполнитель реализован как асинхронный контекстный менеджер:

```python
async with AsyncTaskExecutor(handlers, worker_count=2) as executor:
    await executor.run(queue)
```

Контекстный менеджер нужен для управления жизненным циклом воркеров и корректной очистки ресурсов даже при ошибках.

### Централизованная обработка ошибок

Ошибки обработчиков перехватываются внутри исполнителя. Информация о сбое записывается в лог, а задача переводится в статус `Failed`.

Для этого список допустимых статусов задачи расширен значением `Failed`.

### Примерные обработчики

Добавлены два обработчика:

- `DefaultTaskHandler` - обрабатывает любую задачу;
- `UrgentTaskHandler` - обрабатывает срочные задачи с высоким приоритетом.

Оба обработчика используют `await asyncio.sleep(...)` как имитацию неблокирующей операции ввода-вывода.

## Использованные темы Python

- `async` / `await`;
- `asyncio.Queue`;
- `asyncio.create_task`;
- `asyncio.to_thread`;
- асинхронные контекстные менеджеры `__aenter__` и `__aexit__`;
- `typing.Protocol` и `@runtime_checkable`;
- централизованное логирование через модуль `logging`;
- расширяемая архитектура на основе контрактов.

## Архитектурный результат

Проект теперь состоит из двух уровней:

- синхронный уровень из лабораторной №3: источники задач и ленивая `TaskQueue`;
- асинхронный уровень лабораторной №4: `AsyncTaskQueue`, `TaskHandler`, `AsyncTaskExecutor`.

Такой подход сохраняет уже реализованную функциональность и добавляет асинхронную обработку без переписывания старого кода.
